---------------- What works end-to-end ---------------------------

Confirmed by e2e-real.ps1 against real MiniMax-M2.7:

  1. Spawn platform agent with opencode provider = minimax-coding-plan,
     model = MiniMax-M2.7.
  2. Platform creates opencode session, binds it to the pool Instance.
  3. Platform broadcasts a TASK_ASSIGN event via Redis directed queue.
  4. Pool consumer tick (2s) drains queue, injects as prompt_async with
     the right provider/model pair.
  5. Opencode reaches MiniMax and writes an assistant turn back to the
     session. Observed real reply: "PONG".
  6. Operator-or-timer triggers Sleep. Pool calls CreateArchiveSession
     as a recovery point, terminates opencode subprocess, flips status
     to dormant. Port released. Instance + session metadata preserved.
  7. New broadcast arrives while dormant (tested by direct Redis RPUSH,
     bypassing BroadcastDirected, to prove the consumer-side path).
  8. Next consumer tick sees status=dormant + PendingCount>0, fires
     EnsureReadyByAgentID. Wake re-spawns opencode on a fresh port,
     creates a fresh session, flips status back to ready.
  9. Next tick drains the buffered broadcast into the new session.
     LLM replies again. Real reply: "WOKEUP".
 10. /agentpool/metrics/:id shows three events in correct chronological
     order: spawn_ready, dormancy, wake.

---------------- Lessons worth keeping ---------------------------

Mistake log during this sprint — things that cost time and the
fix that stuck:

1. exec.CommandContext(ctx, ...) with gin's request ctx kills the
   subprocess the moment the HTTP response returns. Use
   context.Background() for subprocess lifetimes; let your own
   Terminate path own the kill.

2. Windows reserves port ranges for Hyper-V / WSL dynamic alloc.
   On the dev box that was 4091-4290 — right where the pool
   defaulted. Bun's listen() fails with "Failed to start server
   on port N" and opencode exits with code 1 within 2 seconds.
   Check `netsh interface ipv4 show excludedportrange protocol=tcp`
   before picking a pool range. 5500-5599 is clean.

3. Opencode 1.14.21 silently returns parts=0 (empty assistant turn)
   when /session/:id/message is posted without a `model` field.
   Not an error. Easy to spend hours staring at "the prompt went
   in but no reply came back" before realising. Always pass
   { provider, model } explicitly; reject empty-model requests
   at the injector boundary.

4. The UI's /opencode/providers endpoint surfaces LLMEndpoint
   rows (`llm_xxxx`), not opencode's own provider catalogue.
   Pool agents need the latter. If you auto-map between them you
   will eventually ship the wrong id to a spawn. Separate
   endpoint, separate dropdown.

5. Phase-4 auto-wake at the producer site (BroadcastDirected)
   misses every caller that writes directly to Redis. Move the
   detection to the consumer tick — it scans dormant instances
   and peeks queue length, so auto-wake works regardless of how
   the event got queued.

6. If an event is recorded after a slow shutdown primitive,
   downstream consumers can see the new state and record their
   OWN events before yours lands. Effect: ring trail shows
   dormancy AFTER wake. Fix: record the event at the status
   transition moment, not at the end of the cleanup work.

7. docker exec ... redis-cli RPUSH "key" "{...json...}" strips
   quotes via PowerShell+shell argument parsing. Resulting Redis
   value is corrupt JSON and the consumer's Unmarshal silently
   skips it — the queue drains but no events reach the injector.
   Use `cat payload.json | redis-cli -x RPUSH key` with the
   payload staged via `docker cp`.

---------------- Multi-agent parallel run (1a2b105) ---------------

3-agent real-project E2E (e2e-multi.ps1), confirmed against live
MiniMax-M2.7:

  - spawned pool_a85e7b34 (util-string),
            pool_9cff7783 (util-math),
            pool_b0d70789 (util-datetime)
  - each got its own broadcast → opencode prompt_async →
    real LLM inference → tool calls to write module + test files
  - all three workdirs stayed disjoint (data/pool/<id>/)
  - host-side pytest PASSES on each workdir's tests
  - delivered working code — euclid-algorithm gcd, case/punct-normalising
    palindrome, fromisoformat date parser — not AI-slop stubs

Three bugs surfaced and fixed during this run:

1. Silent drop on BOM-prefixed Redis payload. json.Unmarshal rejects
   UTF-8 BOM; consumer's FetchEvents caught it + continue'd without
   log. PowerShell's Set-Content -Encoding UTF8 writes BOM. Now we
   (a) TrimPrefix "\ufeff" before unmarshal, (b) log any remaining
   parse failure with preview. The e2e script switched to
   [System.IO.File]::WriteAllBytes.

2. Heartbeat checker offlining pool agents. The MCP-heartbeat sweep
   ran on ALL online agents including platform-hosted ones, which
   never heartbeat via the poller. At the 7-minute mark every pool
   agent got DB-flipped to offline while still serving requests.
   Now scoped out via WHERE is_platform_hosted = false.

3. Windows cmd '>' redirection fully buffers stdio to files. Go's
   log.Printf writes appeared 4+ minutes late. Not a code fix —
   just documented in progress.txt so the next debugger doesn't
   chase a phantom bug.
