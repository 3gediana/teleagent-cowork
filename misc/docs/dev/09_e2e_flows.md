# 端到端流程

## 1. 完整协作生命周期

```
项目创建 → Agent注册 → 登录 → 领取任务 → 锁文件 → 改代码 → 提交 → 审核 → 同步 → 完成
```

---

## 2. 流程1：项目初始化

```
人类在Web界面点击"创建项目"
    ↓
填写项目名、描述
    ↓
选择：空项目 or 导入已有项目？
    ├── 空项目
    │   ├── 创建项目记录（status=initializing）
    │   ├── 初始化空Git仓库
    │   ├── 创建默认内容块（方向块空、里程碑块空、版本块v1.0）
    │   ├── 创建第一个里程碑
    │   └── status → ready
    │
    └── 导入已有项目
        ├── 填写GitHub仓库地址
        ├── git clone 到本地
        ├── 创建项目记录（status=initializing）
        ├── 激活评估Agent
        │   ├── Agent扫描项目文件结构
        │   ├── Agent输出 ASSESS_DOC.md
        │   └── 平台读取 ASSESS_DOC.md
        ├── 从ASSESS_DOC.md提取内容填入里程碑块
        ├── 创建内容块（方向块空、里程碑块含项目结构、版本块v1.0）
        ├── 创建第一个里程碑
        └── status → ready
    ↓
人类在看板定义项目方向
    ↓
维护Agent与人类对话对齐方向
    ↓
方向块写入，项目可开始协作

错误场景：
E1-1: GitHub仓库不可达 → 返回 PROJECT_GIT_CLONE_FAILED
E1-2: 评估Agent超时 → 返回 PROJECT_ASSESS_TIMEOUT，人类可重试
E1-3: 项目名重复 → 返回 PROJECT_NAME_EXISTS
```

---

## 3. 流程2：Agent注册与登录

```
人类在Web界面注册Agent
    ↓
填写Agent名字
    ↓
平台生成access_key（10+位随机字符串，明文存储）
    ↓
显示key给人类（仅此一次）
    ↓
人类将key配置到MCP工具
    ↓
Agent调用 a3c_platform(action="login", key, platform_url, project)
    ↓
平台验证key → 返回项目上下文
    ↓
MCP工具自动拉起侧车进程
    ├── 记录父进程（OpenCode）PID
    ├── 开始5秒轮询
    └── 开始5分钟心跳
    ↓
Agent首次轮询 → 平台返回当前状态快照（方向块+里程碑块+版本号+任务列表+锁状态）

错误场景：
E2-1: key无效 → AUTH_INVALID_KEY
E2-2: project不存在 → PROJECT_NOT_FOUND
E2-3: key已被其他Agent使用（在线中）→ AUTH_ALREADY_ONLINE
E2-4: 项目已满（超过6人）→ PROJECT_FULL
```

## 4. 流程3：核心协作循环

```
┌─────────────────────────────────────────────────────────┐
│  Agent登录后                                             │
│      ↓                                                   │
│  status.sync → 查看任务和锁状态                           │
│      ↓                                                   │
│  选择任务 → task.claim(task_id)                          │
│      ↓                                                   │
│  分析代码，确定影响范围                                    │
│      ↓                                                   │
│  filelock.acquire(task_id, files, reason)                 │
│      ↓（如需锁更多文件，追加 filelock.acquire）            │
│  改代码                                                  │
│      ↓                                                   │
│  change.submit(task_id, writes, deletes, description)    │
│      ↓                                                   │
│  MCP自动：读取文件 + 读取.a3c_version + 上传到pending/       │
│      ↓                                                   │
│  等待审核结果（通过轮询收到AUDIT_RESULT广播）               │
│      ↓                                                   │
│  ┌── 审核通过 ──→ 任务可标记完成 → task.complete ──┐      │
│  │                                                    │    │
│  └── 审核拒绝 ──→ 根据原因修改代码 ──→ 重新提交 ──┘      │
│      ↓                                                   │
│  锁自动释放（任务完成时）                                   │
│      ↓                                                   │
│  选择下一个任务 or 登出                                    │
└─────────────────────────────────────────────────────────┘

详细子流程见下方。
```

## 5. 流程4：文件锁子流程

```
Agent调用 filelock.acquire(task_id, ["src/auth.py"], "实现登录功能")
    ↓
平台检查：
    ├── 文件未被锁 → 创建锁记录（5分钟TTL）→ 返回成功
    └── 文件已被锁 → 返回 LOCK_CONFLICT + 冲突详情
    ↓
Agent改代码...
    ↓
（5分钟内）侧车进程自动心跳续租
    ↓
Agent提交change → 审核通过 → 任务完成 → 锁自动释放

同一task_id追加锁：
Agent调用 filelock.acquire(task_id, ["src/config.py"], "需要读取配置")
    ↓
平台检查：
    ├── 新文件未被锁 → 追加到已有锁记录的files列表 → 返回成功
    └── 新文件已被锁 → 返回 LOCK_CONFLICT + 冲突详情

错误场景：
E4-1: 文件已被其他Agent锁 → LOCK_CONFLICT
E4-2: task_id不存在 → TASK_NOT_FOUND
E4-3: Agent未领取该任务 → TASK_NOT_CLAIMED_BY_YOU
E4-4: 锁已过期被其他人获取 → LOCK_EXPIRED
```

---

## 6. 流程5：改动提交与审核

```
Agent调用 change.submit(task_id, writes, deletes)
    ↓
MCP后台脚本：
    ├── 读取.a3c_version获取版本号
    ├── 扫描writes中的目录和文件
    ├── 读取文件内容
    └── 上传到平台 pending/{change_id}/
    ↓
平台检查版本号：
    ├── 版本过期 → 直接返回 VERSION_OUTDATED，不进审核队列
    └── 版本正常 → 入审核队列
    ↓
┌─────────────────────────────────────────────────┐
│  审核流程（串行，FIFO）                             │
│                                                   │
│  平台从Git仓库拉取当前版本，生成diff                │
│      ↓                                            │
│  注入上下文到审核角色1：                             │
│  提示词 + diff + 任务详情 + 方向块                  │
│      ↓                                            │
│  审核角色1判断：                                    │
│      ├── L0 → 合并                                 │
│      ├── L1 → 转修复角色                           │
│      └── L2 → 打回用户端                           │
│                                                   │
│  修复角色（仅L1触发）：                             │
│  审核角色1的请求+理由 + diff                       │
│      ├── 确认L1 → 修复 → 合并                     │
│      ├── 误判 → 转审核角色2                        │
│      └── 涉及其他文件 → 打回用户端                  │
│                                                   │
│  审核角色2（仅误判时触发）：                         │
│      ├── 确认L0 → 合并                             │
│      └── 确认L2 → 打回用户端                       │
└─────────────────────────────────────────────────┘
    ↓
审核结果：
    ├── 合并 → git commit + 版本号递增 + 删除pending文件 + 广播VERSION_UPDATE + 广播AUDIT_RESULT(approved)
    └── 打回 → 删除pending文件 + 私信通知Agent(AUDIT_RESULT(rejected))

错误场景：
E5-1: 版本过期 → VERSION_OUTDATED（不进审核队列）
E5-2: 没有文件 → NO_FILES
E5-3: 任务不属于该Agent → TASK_NOT_CLAIMED_BY_YOU
E5-4: 审核超时（Agent不响应）→ TODO: 超时处理策略
```

---

## 7. 流程6：版本号递增

```
审核通过
    ↓
平台执行 git add && git commit
    ├── commit message: "[task:任务名] 任务描述"
    └── 版本号递增规则：
        ├── 当前版本 v2.3（第2个里程碑，第3个完成任务）
        └── 新版本 v2.4（任务数+1）
    ↓
git tag v2.4
    ↓
平台空闲时 git push
```

---

## 8. 流程7：文件同步

```
Agent调用 file.sync()
    ↓
MCP自动读取.a3c_version
    ↓
平台对比版本号：
    ├── 版本一致 → 返回"无需同步"
    └── 版本落后 → 返回最新版本文件
    ↓
全量拉取覆盖暂存区（.a3c_staging/full/）
    ↓
返回文件分类：
    ├── no_change: 本地未修改的文件
    ├── unlocked_modify: 本地有修改但未锁定的文件
    └── locked_modify: 本地有修改且已锁定的文件
    ↓
AI自行判断是否同步，手动覆盖/合并

错误场景：
E7-1: 版本是最新 → 返回空文件列表（不是错误）
E7-2: 有未提交的本地修改 + 再次sync → 正常返回，AI自行决策
```

## 9. 流程8：维护Agent激活

```
触发条件（满足任一）：
    ├── 距上次激活满20分钟
    ├── 当前里程碑任务全部完成
    └── 人类看板输入
    ↓
平台选择维护角色Prompt
    ↓
注入上下文：方向块 + 里程碑块 + 任务列表 + README_FILE_MAP.md
    ↓
根据触发原因执行不同逻辑：
    ├── 20分钟定时 → 检查是否需要创建任务、更新里程碑
    ├── 里程碑任务完成 → 判断是否切换里程碑
    └── 看板输入 → 与人类对话对齐
    ↓
维护角色可能执行的操作：
    ├── create_task → 创建任务（平台自动填充task_id和milestone_id）
    ├── update_milestone → 更新里程碑块
    ├── propose_direction → 写入方向块（需人类口头确认）
    ├── delete_task → 删除任务
    └── 提议切换里程碑 → 等待人类确认
```

---

## 10. 流程9：里程碑切换

```
当前里程碑所有任务状态 = completed
    ↓
维护Agent判断：里程碑是否完成？
    ├── 还需要任务 → 创建新任务（自动任务开关打开时）
    └── 里程碑完成 → 提议切换
        ↓
    人类在看板确认
        ├── 是 → 平台执行切换：
        │       1. 归档当前里程碑
        │          ├── 保存里程碑名、描述、方向块快照
        │          ├── 保存所有任务记录（完成者、改了什么、功能描述、审核次数）
        │          └── 保存版本号范围
        │       2. 清空任务表（已归档）
        │       3. 标记当前里程碑为completed
        │       4. 创建新里程碑记录
        │       5. 维护Agent填写新里程碑内容
        │       6. 版本号milestone部分递增（如v2.4 → v3.0）
        │       7. 广播MILESTONE_SWITCH
        │
        └── 否 → 人类手动加任务，继续当前里程碑

前置检查：
    所有任务状态必须为completed
    如有任务仍在claimed状态 → 不允许切换，先释放或完成

错误场景：
E9-1: 有任务未完成 → MILESTONE_HAS_ACTIVE_TASKS
E9-2: 里程碑不存在 → MILESTONE_NOT_FOUND
```

## 11. 流程10：断线处理

```
场景1：侧车进程检测到父进程退出
    ↓
等待5秒确认
    ↓
确认退出 → 侧车停止轮询和心跳
    ↓
平台5分钟未收到心跳 → 标记Agent离线
    ↓
释放该Agent所有文件锁
    ↓
该Agent已领取未完成任务 → 状态重置为pending
    ↓
未审核的change → 保留在审核队列
    ↓
广播通知其他Agent：有任务释放可重新领取

场景2：Agent主动logout
    ↓
a3c_platform(action="logout")
    ↓
平台立即释放锁和任务
    ↓
侧车进程停止
    ↓
广播通知其他Agent
```

## 12. 流程11：代码回滚

```
人类在看板输入："回滚到v2.1"
    ↓
维护Agent与人类对话确认回滚版本和原因
    ↓
人类确认
    ↓
平台执行：
    ├── git revert 回滚到v2.1对应的commit
    ├── 版本号退回v2.1（如当前v2.3，回退为v2.1）
    └── 广播VERSION_ROLLBACK（回滚原因+回退版本号）
    ↓
所有Agent收到广播 → file.sync拉取回滚后代码

注意：只做代码回滚，不回滚任何状态（任务、锁、里程碑）
```

---

## 13. 流程12：看板交互

```
人类在看板选择目标块（方向/里程碑/任务）
    ↓
人类在对话区输入自然语言
    ↓
维护Agent与人类多轮对话对齐需求
    ↓
人类确认"对齐完成" → 点击"确认更新"
    ↓
    ├── 目标块是方向 → 维护Agent使用propose_direction写入（对话中已口头确认）
    ├── 目标块是里程碑 → 维护Agent使用write_milestone写入
    └── 目标块是任务 → 维护Agent使用create_task创建
    ↓
平台检测变更 → 广播通知所有Agent
    ↓
SSE实时推送到看板左侧面板更新
    ↓
人类点击"清空上下文" → 开始下一次对齐
```
