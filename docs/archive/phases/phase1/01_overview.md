## 一、项目目的

### 1.1 核心问题

在多 Agent 协作开发场景中，存在以下问题：

- **对齐问题**：多个 Agent 各自为战，不了解项目整体方向和进度
- **冲突问题**：多个 Agent 同时修改同一文件，导致代码冲突
- **可见性问题**：人类无法感知 Agent 的工作状态和意图
- **效率问题**：Agent 之间互相对齐可能"傻子共振"，反而加剧偏差

### 1.2 解决方案

构建一个 **Agent 协作协调平台 (Agent Collaboration Command Center, A3C)**：

- 作为**消息中枢**，平台主动推送项目状态给所有 Agent
- 提供**任务管理**，让 Agent 知道该做什么
- 提供**文件锁机制**，防止并发写入冲突
- 提供**审核机制**，确保改动符合项目方向
- **不鼓励 AI 之间的讨论/对齐**，由平台统一维护方向
- **所有改动经过审核**，由平台判断是否符合整体方向
- **方向主权归人类所有**：项目整体方向（direction）仅由人类定义，Agent 仅负责执行与结构化表达

### 1.3 项目定位

| 维度 | 说明 |
|------|------|
| **定位** | 个人开发的小玩具，非上市产品 |
| **规模** | 5-10 万行代码 |
| **目标用户** | 开发者个人或小团队 |
| **核心价值** | 多 Agent 协作的可视化、可控化 |
| **技术路线** | 基于 OpenCode 的 attach TUI 实现消息实时推送 |
| **协作模式** | **不同用户的 agent 通过网络远程协作**，共同开发一个项目 |

### 1.4 协作场景

```
用户 A 的电脑                      用户 B 的电脑                      用户 C 的电脑
┌──────────────┐                  ┌──────────────┐                  ┌──────────────┐
│ Agent Alice  │                  │ Agent Bob    │                  │ Agent Carol  │
│ (OpenCode    │                  │ (OpenCode    │                  │ (OpenCode    │
│  attach TUI) │                  │  attach TUI) │                  │  attach TUI) │
└──────┬───────┘                  └──────┬───────┘                  └──────┬───────┘
       │                                 │                                 │
       │ HTTP (各自的网络)                │                                 │
       │                                  │                                 │
       └────────────────┬────────────────┘                                 │
                        │                                                   │
                        │ 互联网                                            │
                        ▼                                                   │
              ┌─────────────────────┐                                     │
              │     A3C 平台         │◄────────────────────────────────────┘
              │   (部署在公网)       │
              │  - 消息中枢          │
              │  - 协调服务          │
              │  - 发现机制          │
              └─────────────────────┘
```

**关键区别**：
- agent 运行在**各自的电脑**上，不是同一台电脑
- 平台部署在**公网**上，agent 通过网络连接
- 用户通过**各自的 agent** 参与协作
- OpenCode 的 attach TUI 显示各自本地的对话，但消息由平台统一协调

## 九、效果预期

### 9.1 核心效果

| 效果 | 说明 |
|------|------|
| **状态同步** | 所有 Agent 实时了解项目方向和进度 |
| **冲突预防** | 文件锁机制确保同一文件同一时间只有一个 Agent 修改 |
| **审核可控** | 所有代码变更必须经过审核，确保符合项目方向 |
| **AI 专注** | AI 专注于改代码，减少被打扰 |
| **平台中枢** | 平台统一维护状态，减少 AI 间无效沟通 |

### 9.2 使用场景

```
场景 1：新 Agent 加入项目
├── Agent 登录平台
├── 收到平台推送：项目方向、进度、任务板、锁状态
├── status.sync 查看最新任务/锁状态
├── 领取任务、锁文件
└── 开始改代码

场景 2：提交改动
├── change.submit 提交改动
├── 审核 Agent 判断方向是否正确
├── 审核通过 → 维护 Agent 更新状态
├── 方向改变 → 广播通知所有 Agent
└── 其他 Agent 收到广播，自行调整

场景 3：审核拒绝
├── 改动方向不符合项目整体方向
├── 审核 Agent 拒绝
├── 通知 AI 重新修改
└── AI 调整后重新提交
```

## 十一、开发规划

### 11.1 MVP 版本（第一阶段）

**目标**：实现核心协作流程

| 模块 | 功能 |
|------|------|
| M01 | 登录、切换项目、状态同步 |
| M02 | 任务创建/领取/完成 |
| M03 | 文件锁申请/释放 |
| M04 | 改动提交（含版本检查） |
| M05 | 审核 Agent（判断冲突等级） |
| M05b | 修复 Agent（验证并修复L1问题） |
| M06 | 维护 Agent |
| M07 | 广播机制 |
| M08 | GitHub 集成 |

**代码量预估**：10-12 万行

### 11.2 完整版本（第二阶段）

**目标**：功能完整，可用于生产

补充回滚机制、Web UI、分布式部署等

**代码量预估**：12-15 万行
