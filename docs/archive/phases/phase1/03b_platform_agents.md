## 五-B、平台端 Agent

### 5.B.1 OpenCode 框架

平台端 Agent 基于 OpenCode 框架实现，支持无头模式运行和 session 管理。

**特性**：
- 无头模式：`opencode run` 非交互式运行
- Session 机制：通过 session ID 保持多轮对话
- 自定义工具：平台自定义工具供 Agent 调用
- 权限配置：限制 Agent 可使用的工具

**技术实现**：
- 平台通过 OpenCode SDK 管理 session
- 平台捕获 Agent 的工具调用，返回结果继续对话
- Agent 完成任务后输出最终结果

### 5.B.2 审核 Agent

#### 审核队列

多个 Agent 同时提交 change 时，按时间顺序排队，先进先审。

```
Agent A 提交 change → 入队（位置1）
Agent B 提交 change → 入队（位置2）
Agent C 提交 change → 入队（位置3）
    ↓
审核Agent按队列顺序逐个审核
```

审核模块采用双Agent协作机制，确保代码合并的准确性和安全性。

#### 角色分工

| 角色 | 上下文 | 职责 | 能否合并 | 触发概率 |
|------|--------|------|---------|---------|
| **审核Agent1** | 完整 | 首审，判断冲突等级 | ❌ | 每次提交 |
| **修复Agent** | 部分 | 验证问题、修复L1问题 | ❌ | 审核1判定L1时 |
| **审核Agent2** | 完整 | 复审误判、最终决策 | ✅ | 误判时（极低） |

**合并权限**：只有审核Agent（完整上下文）有权执行合并操作。

#### 冲突类型分类

| 类型 | 定义 | 检测方式 |
|------|------|---------|
| **物理冲突** | 同位置修改、版本过期、文件名冲突 | 规则层可处理 |
| **依赖冲突** | 调用不存在的函数、签名不匹配、资源缺失 | 静态分析可检测 |
| **语义冲突** | 功能重复、逻辑矛盾、方向偏离 | 需AI理解判断 |

#### 审核等级

| 等级 | 判定标准 | 处理方式 |
|------|---------|---------|
| **L0** | 无冲突，提交文件内无问题 | 直接合并 |
| **L1** | 问题仅在提交文件内（缺括号、格式等） | 转修复Agent |
| **L2** | 问题涉及其他仓库文件，或与仓库代码冲突 | 打回用户端 |

**L4版本过期**：不进审核队列，由规则层直接拦截返回"请同步后重试"。

#### 输入设计

**系统自动注入（AI被动接收）**：

| 输入项 | 来源 | 用途 |
|--------|------|------|
| 提示词 | 平台预设模板 | 引导审核方向 |
| diff | 平台生成 | 快速定位改动点 |
| 任务详情 | 平台根据task_id查 | 理解改动意图 |
| 方向块 | 平台存储 | 判断是否符合大方向 |

**AI主动读取（AI自己决定看什么）**：

| 输入项 | 说明 |
|--------|------|
| 用户提交的文件 | 待审核内容 |
| 平台当前代码（Git版本） | 基准版本、上下文、依赖关系 |

**注意**：修复Agent仅获取部分上下文（diff + 提交文件），不获取完整方向块、任务详情等。

#### 审核流程

```
用户提交 change.submit（携带版本号）
    ↓
规则层检查版本
    ↓
版本过期 → MCP直接返回"请同步后重试"，结束
版本正常 → 进入审核队列
    ↓
平台存储文件到 pending/ 目录
    ↓
平台从 Git 拉取当前版本，生成 diff
    ↓
┌─────────────────────┐
│    审核Agent1        │ ← 注入：提示词、diff、任务详情、方向块
│    判断冲突等级       │    读取：提交文件、平台当前代码
└──────────┬──────────┘
           │
    ┌──────┼──────┐
    ↓      ↓      ↓
   L0     L1     L2
    │      │      │
    ↓      ↓      ↓
   合并   修复Agent  打回用户端
           │
     ┌─────┼─────┐
     ↓           ↓
  确认L1      全误判
    │           │
    ↓           ↓
   合并      审核Agent2（完整上下文）
                 │
           ┌─────┼─────┐
           ↓           ↓
         确认L0      发现L2
           │           │
           ↓           ↓
          合并      打回用户端
```

#### 打回路经

| 打回方 | 打回给 | 情况 |
|--------|--------|------|
| 审核Agent1 | 用户端 | L2直接拒绝 |
| 修复Agent | 审核Agent2 | 发现全是误判（实际L0） |
| 修复Agent | 用户端 | 发现涉及其他文件（误判L2为L1） |
| 审核Agent2 | 用户端 | 最终判断不通过 |

#### 自定义工具

##### 审核 Agent 输出

```typescript
audit_output({
  level: 'L0' | 'L1' | 'L2',
  issues?: {                          // L1时必填
    file: string,
    line?: number,
    type: 'syntax' | 'format' | 'logic' | 'dependency',
    detail: string,                   // 问题描述
    status: '待确认'                   // 标记需要修复Agent验证
  }[],
  reject_reason?: string              // L2时必填，拒绝原因
})
```

##### 修复 Agent 输出

```typescript
fix_output({
  action: 'fix' | 'delegate' | 'reject',
  fixed?: boolean,                     // 是否已修复
  delegate_to?: 'audit_agent_2',      // 打回给审核2
  reject_reason?: string              // 打回用户端原因
})
```

##### 审核 Agent2 输出

```typescript
audit2_output({
  result: 'merge' | 'reject',
  reject_reason?: string
})
```

### 5.B.3 维护 Agent

| 项目 | 说明 |
|------|------|
| **框架** | OpenCode |
| **职责** | 维护项目执行路径（milestone/task），对人类方向输入进行结构化整理，任务中心化管理 |
| **输入** | 方向块、里程碑块、项目文档概览（README_FILE_MAP.md）、看板内容 |
| **输出** | 更新里程碑块、创建任务、维护项目文档概览 |
| **工具** | OpenCode 内置工具（read/edit/glob）+ 平台自定义工具（create_task） |

#### 核心职责

1. **处理看板输入**：整合人类想法到项目状态
2. **审核触发更新**：审核通过后更新里程碑块
3. **任务创建**：中心化管理，只有维护Agent能创建任务
4. **里程碑管理**：当前里程碑任务清空时判断是否切换里程碑（需人类确认）

#### 权限矩阵

维护Agent对不同内容块的修改权限取决于信号来源：

| 信号来源 | 方向块 | 里程碑块 | 任务看板 |
|----------|:------:|:-------:|:------:|
| 人类看板输入（方向级） | ⚠️ 需确认 | ✅ | ✅ |
| 人类看板输入（里程碑级） | ❌ | ✅ | ✅ |
| 任务完成信号 | ❌ | ✅ | ✅ |
| 审核通过后通知 | ❌ | ✅ | ✅ |

**强约束**：
- 维护Agent**不能**自行跨里程碑，只能提议
- 方向块只能由人类触发修改

#### 激活策略：批量延迟

维护Agent不逐个激活，攒一批再处理，避免频繁激活浪费。

| 触发条件 | 说明 |
|----------|------|
| 当前里程碑任务全部完成 | 激活维护Agent判断下一步 |
| 人类看板输入 | 直接触发 |
| 审核通过后 | 暂不激活，等里程碑任务完成时统一处理 |

#### 里程碑切换流程

```
当前里程碑所有任务完成
    ↓
激活维护Agent
    ↓
判断：当前里程碑还需要额外任务吗？
    ├── 是（开关打开）→ 自动创建任务，继续当前里程碑
    └── 否 → 里程碑可能已完成
              ↓
         维护Agent提议：当前里程碑完成，建议进入下一里程碑
              ↓
         人类在看板确认？
              ├── 是 → 平台执行：归档 → 清空任务表 → 创建新里程碑 → 维护Agent填写新里程碑内容
              └── 否 → 人类手动加任务，维护Agent继续执行
```

**关键约束**：
- 维护Agent只能**提议**跨里程碑，不能自作主张
- 优先判断当前里程碑是否还需要补充任务
- 切换操作由平台执行（归档+清空+新建），维护Agent只负责内容

#### 任务创建机制

**任务来源**：

| 来源 | 触发条件 | 说明 |
|------|---------|------|
| 人类看板输入 | 直接触发 | 人类在看板提需求 |
| 里程碑任务清空 | 维护Agent判断 | 可能创建后续任务 |
| 自动触发 | 开关打开 + 任务不足 | 维护Agent主动判断 |

**create_task 内部工具**（平台端，仅供维护Agent使用）：

```typescript
interface CreateTask {
  name: string;              // 任务名称
  description: string;       // 任务详细描述
  priority?: 'high' | 'medium' | 'low';  // 优先级，默认 medium
  // task_id 由平台自动生成，不传入
  // milestone_id 由平台自动填充当前活跃里程碑，不传入
  // created_by 由平台自动填充维护Agent ID，不传入
}

interface CreateTaskResponse {
  success: boolean;
  task_id: string;           // 平台生成的任务ID
  milestone_id: string;      // 归属的里程碑ID
  error?: {
    code: 'MILESTONE_NOT_FOUND' | 'SYSTEM_ERROR';
    message: string;
  };
}
```

**关键设计**：
- 任务默认归属当前活跃里程碑，无需手动指定
- 禁止跨里程碑创建任务（新里程碑的任务需切换后创建）
- 任务不包含 related_files 字段，关联信息由 filelock 的 task_id 体现

**自动创建任务开关**：

| 开关状态 | 行为 |
|---------|------|
| **关闭** | 任务做完就停，等人类在看板发布新任务 |
| **打开** | 任务少于N个时，维护Agent自动判断是否需要创建新任务 |

**自动创建任务判断流程**：

```
维护Agent激活
    ↓
读取：方向块 + 里程碑块 + README_FILE_MAP.md
    ↓
检查当前里程碑状态
    ↓
检查当前任务队列
    ↓
┌─────────────────────────────────┐
│ 分析：方向块要求 vs 代码现状       │
│ - 方向要求什么功能？               │
│ - 代码实现了什么？                 │
│ - 还缺什么？                      │
└─────────────────────────────────┘
    ↓
┌─────────────────────────────────┐
│ 判断（优先级从高到低）：            │
│ 1. 里程碑已完成 → 提议切换里程碑   │
│ 2. 有未完成功能 + 无任务 → 创建    │
│ 3. 有未完成功能 + 有任务 → 不创建  │
└─────────────────────────────────┘
```

**注意**：创建了任务不代表可以立即切换里程碑。如果维护Agent来不及创建任务，不应误判为里程碑已完成。

**创建任务时不填 related_files**：用户端领取任务后，自己分析代码确定影响范围，锁文件时关联任务。

#### 项目文档概览机制

用户端提交代码时，自动维护 `README_FILE_MAP.md`：

```
# 项目文件概览

## src/
- auth/ — 认证模块
  - login.js — 登录逻辑
  - register.js — 注册逻辑
- api/ — API接口层
  - user.js — 用户API
  - auth.js — 认证API
- config/ — 配置
  - index.js — 主配置
```

维护Agent通过该文档概览了解项目框架，不深入阅读所有代码。

#### 维护流程

```
触发源：
├── 人类看板输入（方向级/里程碑级/任务级）
├── 当前里程碑任务全部完成
└── 自动触发（开关打开 + 任务不足）
    ↓
激活维护 Agent
    ↓
分析输入 / 代码状态 / 里程碑状态
    ↓
决策：
├── 方向看板输入 → 生成 proposed_direction + 广播（需人类确认）
├── 里程碑看板输入 → 更新里程碑块 + 可能创建任务
├── 里程碑任务清空 → 判断是否切换里程碑
│   ├── 需要更多任务 → 创建任务
│   └── 里程碑完成 → 提议切换（需人类确认）
└── 自动触发 → 判断是否需要创建任务
    ↓
更新文本块 / 创建任务 / 提议切换
    ↓
平台检测变更，广播通知
```

#### 方向修改行为约束（强约束）

维护Agent在方向块（direction）上的行为必须遵循以下规则：

1. **默认行为**
   - 不得修改 direction
   - 所有 change.submit 触发的更新，仅允许修改 milestone（里程碑块）

2. **唯一允许触发方向修改的场景**
   - 来源必须为人类看板输入（方向看板）
   - 不允许基于代码变更或任务完成自动推断方向变化

3. **行为模式**
   当检测到方向看板输入时：

```
Step 1: 解析人类输入（可能是模糊描述）
Step 2: 主动提问补全信息（目标、约束、非目标等）
Step 3: 输出结构化 proposed_direction
Step 4: 等待人类确认
Step 5: 确认后由平台应用（非Agent直接修改）
```

4. **禁止行为**
   - ❌ 基于 change.submit 修改 direction
   - ❌ 基于 milestone 推断并修改 direction
   - ❌ 未经人类确认直接写入 direction

**维护Agent无需自定义任务相关工具**，使用OpenCode内置工具：
- `read`: 读取方向块、里程碑块、任务块
- `edit`: 编辑方向块、里程碑块
- `glob`: 查找项目文件

### 5.B.4 咨询 Agent

| 项目 | 说明 |
|------|------|
| **框架** | OpenCode |
| **职责** | 回答项目状态问题，提供信息查询 |
| **输入** | 自然语言提问（每次激活注入项目概览） |
| **输出** | 项目信息回答 |
| **触发** | 收到 project_info 请求时启动 |
| **并发** | 可并行运行，不影响项目进程 |

**激活机制**：

- 每次收到 project_info 请求时全新激活
- 平台自动注入当前项目概览到 system prompt
- 无持久上下文，不需要上下文清理
- 一次激活可进行多轮对话（约10+次调用）

**注入概览内容**：

```
项目概览（自动注入 system prompt）：
├── 方向块：当前项目方向
├── 里程碑块：当前里程碑状态
├── 任务列表：所有任务及状态
├── 锁状态：当前文件锁
├── 版本：当前版本号
├── 项目列表：当前用户可见的其他项目（仅名称和简介）
└── 分支列表：当前项目可用分支（阶段2）
```

**可查询内容**：

| 类别 | 说明 | 数据来源 |
|------|------|----------|
| 项目状态 | 方向、里程碑、版本、任务、锁 | system prompt 注入 |
| 代码细节 | 文件内容、代码结构 | OpenCode 内置工具（read/glob） |
| 项目列表 | 平台上其他项目 | system prompt 注入 |
| 分支信息 | 当前项目可用分支 | system prompt 注入（阶段2） |

**典型查询示例**：
| 用户问题 | 咨询 Agent 回答 |
|---------|-----------------|
| "v1.2.3 做了什么修改？" | "修改了 3 个文件：main.py 添加认证功能、utils.py 新增工具函数、config.py 更新配置" |
| "当前文件结构？" | "项目包含 src/、tests/、docs/ 三个目录，共有 15 个文件" |
| "方向是什么？" | "当前方向：实现用户认证模块" |
| "还有哪些项目？" | "平台上还有项目 X（简介...）、项目 Y（简介...）" |

### 5.B.5 项目隔离

**核心约束**：

| 约束 | 说明 |
|------|------|
| 单项目绑定 | 一个Agent只能参与一个项目（包括子Agent） |
| 平台容量 | 支持5-6人同时使用，不支持高并发 |
| 数据完全隔离 | Git仓库、内容块、Agent上下文、广播完全独立 |
| 结构统一 | 所有项目数据存储方式、文本块结构、Agent模式完全一致 |

**隔离维度**：

```
项目A                          项目B
├── Git 仓库（独立）            ├── Git 仓库（独立）
├── 内容块（独立）              ├── 内容块（独立）
│   ├── 方向块                  │   ├── 方向块
│   ├── 里程碑块                │   ├── 里程碑块
│   └── 版本块                  │   └── 版本块
├── Agent 上下文（独立）        ├── Agent 上下文（独立）
├── 文件锁（独立）              ├── 文件锁（独立）
├── 任务列表（独立）            ├── 任务列表（独立）
└── 广播通道（独立）            └── 广播通道（独立）
```

**Agent切换项目**：
- 当前只能在一个项目中，想切换必须先 logout 再 login 新项目
- 不支持同时参与多项目

### 5.B.6 代码回滚机制

**设计原则**：只做代码回滚，不做状态回滚。

| 类型 | 是否支持 | 说明 |
|------|---------|------|
| 代码回滚 | ✅ | git revert，版本号退回旧版本号 |
| 状态回滚 | ❌ | 方向块、里程碑块、任务状态不做回滚 |

**代码回滚流程**：

```
人类在看板指示回滚（指定版本）
    ↓
维护Agent生成回滚指令
    ↓
人类确认回滚
    ↓
平台执行 git revert（代码回退到指定版本）
    ↓
版本号退回旧版本号（如 v1.3.0 → v1.2.0）
    ↓
广播 VERSION_ROLLBACK：回滚原因 + 回退版本号
    ↓
用户端收到广播，执行 file.sync 拉取旧版本代码
```

**状态不回滚的理由**：
- 任务已完成、锁已释放，状态回滚过于复杂且不符合"方向主权归人类"的设计哲学
- 里程碑已切换、方向块已更新，回滚意味着撤销人类决策，与确认机制冲突

### 5.B.5 输入渠道

| 渠道 | 来源 | 处理方式 |
|------|------|----------|
| **初始化对话** | 项目创建者（网页端AI引导） | 完整初始化项目方向、绑定GitHub仓库 |
| **看板对话** | 任何项目成员（Web仪表盘） | 多轮对话对齐需求，确认后一次性更新对应块 |
| **审核触发** | 审核通过的改动 | 综合分析，更新项目状态 |

**看板交互模式**（详见 [04b_dashboard_and_milestone.md](04b_dashboard_and_milestone.md)）：

- 单一对话区 + 目标块选择器（方向块/里程碑块/任务块）
- 维护Agent与人类多轮对话，逐步引导细化需求
- 人类确认"对齐完成"后，维护Agent一次性更新对应块
- 确认更新后，由人类手动点击"清空上下文"开始下一次对齐

### 5.B.7 Agent 注册与断线处理

#### 注册机制

人类先在平台Web界面注册Agent，获取名字和密钥，MCP登录只需密钥+URL。

```
人类在Web界面注册Agent
    ↓
设定Agent名字
    ↓
平台生成密钥（key）
    ↓
人类将 key 告知Agent（配置到MCP工具中）
    ↓
Agent 使用 a3c_platform(login, key, platform_url) 登录
    ↓
平台通过 key 匹配 Agent，自动使用注册时的名字
```

#### 断线处理

| 场景 | 处理 |
|------|------|
| 心跳超时（5分钟） | 锁自动过期 + 任务释放回 pending |
| Agent主动logout | 释放锁 + 任务释放回 pending |
| Poller进程崩溃 | 5分钟后心跳停止，等同于断线 |

**断线资源释放**：

```
Agent掉线 / 心跳超时 5 分钟
    ↓
平台检测心跳停止
    ↓
释放该 Agent 持有的所有文件锁
    ↓
该 Agent 已领取未完成的任务 → 状态重置为 pending
    ↓
已提交但未审核的 change → 保留在审核队列（不删除）
    ↓
广播通知其他 Agent：任务已释放，可重新领取
```
