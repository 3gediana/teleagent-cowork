## 二、技术突破

### 2.1 关键发现

OpenCode 的 `opencode attach` 启动的 TUI **可以实时显示通过 API 发送的消息**，而直接 `opencode` 启动的 TUI 不支持此功能。

### 2.2 架构优势

**突破前**：
- 平台无法主动推送消息
- Agent 只能被动轮询获取信息
- 消息传递延迟高、实时性差

**突破后**：
- 平台不主动推送，任何用户（包括 NAT 后的用户）都能使用。
- 依赖 **独立后台进程 (Poller Sidecar)** 实现轮询：
  - `a3c_platform(login)` 时，MCP 工具拉起一个后台轮询脚本。
  - 脚本每 5 秒通过 HTTP GET 请求平台获取消息，并通过 OpenCode API 注入到本地 attach TUI。
  - `a3c_platform(logout)` 或 OpenCode 父进程退出时，销毁轮询进程。
- 实现接近实时的消息传递，同时保持 MCP 工具的无状态设计。

### 2.3 核心约束（重要）

| 约束项 | 说明 |
|--------|------|
| **平台存储代码** | 平台本地存储 Git 仓库，用户通过 MCP 工具同步代码 |
| **变更上传机制** | 用户通过 change.submit 上传改动文件，平台审核后更新本地仓库 |
| **批量 GitHub 同步** | 平台空闲时批量 push 到 GitHub，减少 API 调用 |
| **用户端不直接访问 GitHub** | 用户不直接 pull/push GitHub，所有同步通过平台完成 |

**用户端文件同步**：
- 用户调用 file.sync 获取最新代码
- 平台返回差异文件，MCP 工具自动处理
- 用户本地修改后通过 change.submit 上传

**文件锁机制**：
- 文件锁是声明机制，预告修改范围，不强制阻止他人修改
- 同一文件可以有多个修改，审核 Agent 负责判断冲突并处理

## 三、系统架构

### 3.1 整体架构

```
用户 A 的电脑                      用户 B 的电脑                      用户 C 的电脑
┌──────────────┐                  ┌──────────────┐                  ┌──────────────┐
│ Agent Alice  │                  │ Agent Bob    │                  │ Agent Carol  │
│ (OpenCode    │                  │ (OpenCode    │                  │ (OpenCode    │
│  attach TUI) │                  │  attach TUI) │                  │  attach TUI) │
└──────┬───────┘                  └──────┬───────┘                  └──────┬───────┘
       │ HTTP/HTTPS                      │                                  │
       │                                  │                                  │
       └──────────────────┬───────────────┘                                  │
                         │                                                   │
                         │ 互联网                                             │
                         ▼                                                   │
┌─────────────────────────────────────────────────────────────────────────────┐
│                              A3C 平台 (公网部署)                            │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │  消息广播   │  │  发现服务   │  │  文件锁服务  │  │  审核服务   │        │
│  │  模块       │  │             │  │             │  │             │        │
│  └─────────────┘  └─────────────┘  └─────────────┘  └─────────────┘        │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │  MCP 工具   │  │  私信服务   │  │  协商服务   │  │  任务服务   │        │
│  │  管理       │  │             │  │             │  │             │        │
│  └─────────────┘  └─────────────┘  └─────────────┘  └─────────────┘        │
└─────────────────────────────────────────────────────────────────────────────┘
```

**关键特性**：
- 每个用户的 Agent 运行在**各自的电脑**上，通过 HTTP/HTTPS 连接平台
- 平台部署在**公网**，不主动推送，Agent 轮询 MCP 接口获取消息
- OpenCode attach TUI 显示各自主会话的消息，实现实时可视化
- 文件锁、审核、协商等协调功能由平台统一管理

### 3.2 平台文件存储

平台本地存储 Git 仓库，用户通过 MCP 工具同步代码。

#### 3.2.1 存储结构

```
cache/projects/{project_id}/
├── .git/                       # Git 仓库
├── files/                      # 当前文件（Git 工作目录）
│   ├── src/
│   ├── docs/
│   └── ...
└── pending/                    # 待审核变更（审核后清理）
    ├── change_A/
    │   ├── new/                # 新增文件
    │   │   └── {file_path}
    │   ├── modified/           # 修改文件
    │   │   ├── original/       # 当前版本副本
    │   │   ├── modified/       # 新文件内容
    │   │   └── diff/           # unified diff
    │   └── deleted/            # 删除文件
    │       └── {file_path}
    └── change_B/
        └── ...
```

#### 3.2.2 存储说明

| 目录 | 用途 | 生命周期 |
|------|------|---------|
| `.git/` | Git 仓库，管理版本历史 | 长期存储 |
| `files/` | 当前工作目录 | 长期存储 |
| `pending/{change_id}/` | 待审核变更 | 审核完成后清理 |

#### 3.2.3 关键流程

**审核通过流程**：
```
用户提交 change.submit
  ↓
平台存储到 pending/{change_id}/
  ↓
审核 Agent 审核 diff
  ↓
审核通过 → git add && git commit
  ↓
更新 files/ 目录
  ↓
清理 pending/{change_id}/
  ↓
空闲时批量 git push 到 GitHub
```

**用户同步流程**：
```
用户调用 file.sync({ version: "v1.2.3" })
  ↓
平台对比版本差异
  ↓
返回差异文件列表
  ↓
MCP 工具自动执行操作（覆盖/删除）
  ↓
用户端保存版本号到 .a3c_version
```

```
┌─────────────────────────────────────────────────────────────────┐
│                        消息类型                                  │

│  - 文件锁通知 (FILE_LOCK_NOTIFY)                                │
│  - 任务通知 (TASK_NOTIFY)                                       │
│  - @提及 (MENTION)                                              │
│  - 工作流事件 (WORKFLOW_EVENT)                                  │
├─────────────────────────────────────────────────────────────────┤
│  Agent 消息                                                     │
│  - 私信 (DIRECT_MESSAGE)                                        │
│  - 广播消息                                                     │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      消息分发流程                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 消息产生                                                     │
│     - 平台主动产生（公告、审核结果等）                            │
│     - Agent 发送（私信、广播）                                   │
│                                                                  │
│  2. 消息入队                                                     │
│     - 消息统一存入 Redis 消息队列                                │
│     - 按 target_agent_id 索引                                    │
│                                                                  │
│  3. Agent 轮询获取                                              │
│     - Agent 通过 MCP 工具定期轮询平台                            │
│     - 平台返回该 Agent 的所有未读消息                            │
│     - Agent 通过 OpenCode API 发送到本地                         │
│       POST /session/{id}/message                                 │
│     - Agent attach TUI 实时显示                                  │
│                                                                  │
│  4. 已读确认                                                     │
│     - Agent 调用 mark_read 接口确认收到                          │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

| M08 | **GitHub 集成** | 同步文件树、push 代码 |
