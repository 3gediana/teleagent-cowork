## 六、内容分块与消息广播机制

### 6.1 内容分块

平台将项目状态分成多个独立的文本块存储，每个块由不同的逻辑处理：

#### 6.1.1 广播文本块（3个）

| 块 | 内容 | 更新频率 | 处理方 | 广播 |
|------|------|---------|--------|------|
| **方向块** | 项目背景、目标、技术选型、架构决策 | 低 | 维护 Agent | ✅ |
| **思路块** | 里程碑列表、当前重点、下一步计划、注意事项 | 中 | 维护 Agent | ✅ |
| **版本块** | 最新版本号、修改摘要、修改者、时间 | 中 | 平台 | ✅ |

#### 6.1.2 查询文本块（不广播）

| 块 | 内容 | 查询方式 |
|------|------|---------|
| **任务块** | 任务列表、领取状态、完成状态 | status.sync |
| **锁块** | 锁定的文件、锁定者、锁定时间 | status.sync |

#### 6.1.3 文本块内容示例

**方向块示例**：
```json
{
  "project_name": "A3C 平台",
  "background": "解决多 Agent 协作中的对齐、冲突、可见性问题",
  "goals": ["实现消息中枢", "提供任务管理", "提供审核机制"],
  "tech_stack": ["OpenCode", "MySQL", "Redis", "Git"],
  "architecture": "平台部署在公网，Agent 通过 HTTP 连接"
}
```

**思路块示例**：
```json
{
  "milestones": [
    { "name": "用户认证", "status": "completed" },
    { "name": "数据存储", "status": "in_progress" },
    { "name": "API接口", "status": "pending" }
  ],
  "current_focus": "实现数据存储模块",
  "next_steps": ["完成数据库设计", "实现 MCP 工具"],
  "notes": ["注意并发冲突处理", "确保审核 Agent 判断准确"]
}
```

**版本块示例**：
```json
{
  "version": "v1.2.3",
  "summary": "添加用户认证功能，修复文件锁bug",
  "author": "Alice",
  "timestamp": "2026-04-19T10:00:00Z",
  "files_changed": ["src/auth.py", "src/lock.py"]
}
```

**设计原则**：
- 方向/思路/版本变更必须广播同步
- 任务/锁状态通过查询获取，避免频繁广播
- 版本块只广播最新版本，避免信息过载

### 6.2 广播自动推送机制

**三个广播文本块变更时，自动发送到用户端**：

```
方向块变更 → 平台检测变更 → 广播推送给所有在线 Agent
    ↓
思路块变更 → 平台检测变更 → 广播推送给所有在线 Agent
    ↓
版本块变更 → 审核通过 → 平台更新版本 → 广播推送给所有在线 Agent
```

**广播推送流程**：

```
维护 Agent 更新方向/思路块
    ↓
平台检测到文本块变更
    ↓
平台生成广播消息
    ↓
存入 Redis 消息队列
    ↓
Agent 每 5 秒轮询平台
    ↓
平台返回广播消息
    ↓
Agent 通过 OpenCode API 发送到 attach TUI
    ↓
用户在 TUI 中看到广播消息
```

### 6.2 广播触发时机

| 事件 | 是否广播 | 说明 |
|------|---------|------|
| 方向块变更 | ✅ | 维护 Agent 判断后广播 |
| 思路块变更 | ✅ | 维护 Agent 判断后广播 |
| 版本块变更 | ✅ | 审核通过后广播（只有最新版本） |
| 任务变更 | ❌ | 通过 status.sync 查询 |
| 锁变更 | ❌ | 通过 status.sync 查询 |

### 6.3 消息格式

```typescript
interface BroadcastMessage {
  // 消息头
  header: {
    messageId: string;        // 全局唯一消息ID
    type: 'DIRECTION_CHANGE' | 'THOUGHT_UPDATE' | 'VERSION_UPDATE';  // 广播类型
    version: string;         // 协议版本
    timestamp: string;       // ISO8601 时间戳
  };

  // 消息内容
  payload: {
    block_type: 'direction' | 'thought' | 'version';
    content: string;         // 完整的块内容
    changes?: string;         // 变更摘要（方便 AI 快速理解）
  };

  // 元信息
  meta: {
    agent: 'platform';        // 固定为 platform
    project_id: string;
    triggered_by: string;    // 触发本次更新的 change_id
  };
}
```

### 6.4 广播推送流程

```
审核 Agent 完成审核
    ↓
维护 Agent 被激活
    ↓
综合分析改动内容
    ↓
判断方向/思路是否改变
    ↓
更新对应文本块
    ↓
方向/思路改变 → 广播推送
    ↓
所有在线 Agent 收到广播
    ↓
AI 自行决定是否调整当前工作
```

### 6.5 边界情况处理

| 场景 | 处理方式 |
|------|----------|
| 审核期间新用户上线 | 平台推送完整项目状态快照 |
| 改动审核通过 + 方向改变 | 维护 Agent 更新 + 广播 |
| 方向改变时其他 AI 正在工作 | AI 收到广播后自行检查任务是否还符合 |
| 两人同时领同一任务 | 先到先得，后到收到冲突通知 |

### 6.6 技术实现要点

| 要点 | 说明 |
|------|------|
| **轮询机制** | MCP 工具拉起的 Sidecar 进程每 5 秒后台轮询，获取消息并维持锁心跳 |
| **内容分块存储** | MySQL 按块存储项目状态（方向、进度、任务、锁） |
| **平台主动推送** | 方向/进度变更时，平台主动广播给所有 Agent |
| **Agent 主动查询** | 任务/锁状态通过 status.sync 主动查询 |
| **离线处理** | Redis 队列存储，Agent 上线后首次轮询返回所有离线消息 |
