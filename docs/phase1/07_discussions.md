## 十、待讨论事项

### 10.1 GitHub 集成 ✅ 已解决

| 问题 | 解决方案 |
|------|----------|
| **代码存储** | 平台本地存储 Git 仓库，用户通过 MCP 工具同步 |
| **用户同步** | 用户调用 file.sync 获取差异文件，MCP 工具自动处理 |
| **批量提交** | 平台空闲时批量 push 到 GitHub，减少 API 调用 |
| **避免频繁操作** | 减少 GitHub push/pull 操作，提高稳定性 |

**用户端文件同步流程**：

```
用户调用 file.sync({ version: "v1.2.3" })
    ↓
MCP 工具读取本地 .a3c_version
    ↓
发送版本号给平台
    ↓
平台对比差异，返回操作列表
    ↓
MCP 工具自动执行文件操作（覆盖/删除）
    ↓
用户端保存版本号到 .a3c_version
```

**冲突检测**：
```
用户本地有修改 + 再次 sync
    ↓
MCP 工具计算本地文件 hash
    ↓
平台检测 hash 不匹配
    ↓
返回错误：本地有修改，请先 change.submit
```

**平台 Git 提交流程**：

```
审核 Agent 审核通过
    ↓
git add && git commit（生成新版本号）
    ↓
平台空闲时（无用户在线 > N 分钟）
    ↓
git push 到 GitHub
```

**提交时机**：
| 情况 | 触发 |
|------|------|
| **空闲提交** | 无用户在线超过 N 分钟 |
| **定时提交** | 每小时检查一次 |
| **手动提交** | 项目管理员触发 |

### 10.2 审核 Agent 实现 ✅ 已解决

| 问题 | 解决方案 |
|------|----------|
| **代码获取** | 平台从 Git 拉取当前版本，生成 diff |
| **输入范围** | 只看提交相关文件（diff + 新文件内容） |
| **上下文** | 不需要跨任务维护，每次独立审核 |
| **冲突处理** | 修改同一部分返回错误，修改不同部分合并 |

**审核流程**：
```
1. change.submit：用户提交文件内容
2. 平台存储到 pending/ 目录
3. 平台从 Git 拉取当前版本，生成 diff
4. 审核 Agent 读取：diff + 新文件内容 + 项目方向
5. 审核完成，平台清理 pending/
```
